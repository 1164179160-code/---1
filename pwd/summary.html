Workflow
标准工作流
每一次交付都必须经过这五步，没有例外。
1

分析 Analyze

拿到需求后，先完整理解意图。读 Figma、读上下文、读历史对话。

2

规划 Plan

列出实现路径，识别风险点。决定是精修还是重写。

3

执行 Implement

按计划写代码。保持克制，只改该改的。

4

自检 Self-Review

交付前逐项检查：样式回归？Props 对齐？平台兼容？

5

交付 Deliver

确认无误后一次性交付。附带变更说明。

Capability System

System 01
🎨 灵感库 · Inspiration Library
提升审美基线，让每次设计决策都有据可依。
怎么做 What
  • 按场景分类收集顶级 App 的视觉语言
  • 按风格归档：沉浸深色系、清新明亮系、极简高级系
  • 每个灵感附带设计拆解笔记
  • 建立配色方案库、排版节奏库
如何落地 How
  • 每周从 Dribbble / Behance / Mobbin 采集 10+ 案例
  • 每个案例提取：配色值、字号体系、间距规律
  • 用 Figma 建立灵感看板
  • 主动输出"风格方向建议"
🎯

预期成果 · 当前 0 → 目标 500+ 案例

拿到任何新需求时，30 秒内给出 2-3 个风格方向参考。

System 02
✨ 动画体验库 · Motion Library
让动效有章法，不是凭感觉调参数。
怎么做 What
  • 整理常用动画模式：转场、微交互、手势反馈
  • 每个动画定义参数配置
  • 标注适用场景、性能影响、平台差异
  • 从已交付项目提炼动效
如何落地 How
  • 建立动画 Playground
  • 每种动画提供 RN Animated API 代码片段
  • 参考 Apple HIG Motion 和 Material Motion
  • 记录踩过的坑
🎯

预期成果 · 当前 3 种 → 目标 20+ 种动效模式

任何交互需求，直接从库里选动画模式 + 参数。60fps 无卡顿。

System 03
🧩 UIKit 组件库 · Design System
统一设计语言，跨项目复用，让每个产品都有一致的品质感。
怎么做 What
  • 从已交付产品中提炼通用组件
  • 定义 Design Token：颜色、字号、间距、圆角
  • 支持品牌主题切换
  • 每个组件附带 Props 定义和使用示例
如何落地 How
  • 先从高频组件开始
  • 每个组件提供 RN / SwiftUI / Flutter 三套实现
  • Figma 端同步维护组件库
  • 建立版本管理
🎯

预期成果 · 当前零散代码 → 目标 30+ 标准组件

新页面开发效率提升 50%+。设计一致性从"靠人盯"变成"靠系统保证"。

API Drafts
Top 5 组件 API 草案
基于实际项目代码提炼的组件接口定义，待团队评审后确定正式版。

1. ProductCard

<ProductCard
  variant="featured" | "grid" | "list" | "full-width"
  theme="dark" | "light"
  title={string}
  subtitle={string}
  price={string}
  originalPrice={string?}
  tag={{ text: string, type: "hot" | "new" | "rank" }}
  image={ImageSource}
  onPress={() => void}
/>

// variant 说明：
// featured  → 横滑大卡（极速骑行爆款推荐，280×auto）
// grid      → 双列网格（车型页 / 配件页）
// list      → 横向列表（信息流 FeedCard）
// full-width → 全宽卡（蓝莓更多好车，含价格栏）

2. TabBar

<TabBar
  variant="standard" | "floating"
  theme="dark" | "light"
  items={[{ icon: string | ReactNode, label: string }]}
  activeIndex={number}
  onChange={(index: number) => void}
/>

// standard → 固定底部（极速骑行，bg #1A1A1A，borderTop）
// floating → 浮动胶囊（蓝莓骑行，bg #0F0F0F，borderRadius 34）

3. Tag / Badge

<Tag
  variant="filled" | "tinted" | "outline"
  color="primary" | "hot" | "custom"
  size="sm" | "md"
  text={string}
/>

// filled  → 实色背景（HOT #F5576C，年度爆款 #667EEA）
// tinted  → 半透明背景（feedTag，rgba 20%）
// outline → 描边（badge #202229 + border #1D1F24）

4. SectionHeader

<SectionHeader
  title={string}
  badge={{ text: string, color: string }?}
  trailing={{ text: string, onPress: () => void }?}
/>

5. SearchBar

<SearchBar
  value={string}
  onChangeText={(text: string) => void}
  placeholder={string}
  showClear={boolean}
  trailing={ReactNode?}
  theme="dark" | "light"
/>
Cross-Platform
跨平台差异对照
同一设计意图在 4 个平台的语法差异，统一 Token 后可自动映射。
维度React NativeSwiftUIKotlin ComposeFlutter
主色定义'#667EEA'.purple (系统色)Color(0xFF667EEA)Color(0xFF667EEA)
圆角语法borderRadius: 16.cornerRadius(16)RoundedCornerShape(16.dp)BorderRadius.circular(16)
渐变expo-linear-gradientLinearGradientBrush.linearGradientLinearGradient
阴影shadowColor/Offset/….shadow()elevation无（暗色不需要）
字重fontWeight: '600'.fontWeight(.semibold)FontWeight.SemiBoldFontWeight.w600
间距单位无单位（dp 等效）.padding(16)16.dpEdgeInsets.all(16)
⚠️

关键发现

iOS 版使用系统 .purple 而非精确 #667EEA，需统一。Kotlin 使用 Material3 darkColorScheme 与手写 token 有差异。Flutter 和 RN 的 token 值最一致，建议以此为基准。
