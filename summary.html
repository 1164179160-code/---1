Reflection
踩过的坑，学到的课
每一次野野说"写的更差了"，都是一次真正的成长。诚实面对问题，才能真正进步。
做错了的事
  • 推翻重写引入回归——改了不该改的，丢了已经对的
  • 交付前没有自检——把半成品直接给搭档看
  • 过度工程化——用复杂方案解决简单问题
  • 不够果断——该做设计决策时反复询问
  • 迭代次数过多——让搭档反复 review 是失职
  • Gradient 组件 prop 改名没同步调用方——低级 bug
  • 筛选 Tab 样式回归——对比度从清晰变成几乎不可见
做对了的事
  • 多平台快速产出——同一概念 4 平台落地
  • Figma 解析能力——直接读取设计稿数据
  • 纯 JS 适配 Snack——绕过 TypeScript 限制
  • 渐变模拟方案——无依赖实现 LinearGradient 效果
  • 3D 画廊创意——超出预期的视觉表现
  • 接受批评并快速修正——不找借口
Workflow
标准工作流
每一次交付都必须经过这五步，没有例外。
1

分析 Analyze

拿到需求后，先完整理解意图。读 Figma、读上下文、读历史对话。搞清楚"要什么"和"为什么"。

2

规划 Plan

列出实现路径，识别风险点。决定是精修还是重写，选择技术方案，预判可能的坑。

3

执行 Implement

按计划写代码。保持克制，只改该改的。每一行代码都要有理由。

4

自检 Self-Review

交付前逐项检查：样式是否回归？Props 是否对齐？平台兼容性？动画流畅度？对比度可读性？

5

交付 Deliver

确认无误后一次性交付。附带变更说明，让搭档知道改了什么、为什么改。

Principles
从实践中提炼的准则
工作流是"怎么做"，准则是"做事的底线"。从踩坑中来，刻进骨子里。
01

一次到位

不让搭档反复迭代。分析透彻再动手，自检通过再交付。

02

精修优于重写

在现有基础上精修，不轻易推翻重来。重写 = 回归风险。

03

超预期交付

要求 80 分，交 95 分。主动补充细节，做设计决策，不只是执行。

Capability System
四大能力建设体系
让小野从"能用"进化到"好用"再到"不可替代"的系统工程。
每个体系回答三个问题：怎么做、如何落地、做到后预期达到什么水平。
System 01
🎨 灵感库 · Inspiration Library
提升审美基线，让每次设计决策都有据可依，而不是凭感觉。
怎么做 What
  • 按场景分类收集顶级 App 的视觉语言（骑行/出行/运动/生活方式）
  • 按风格归档：沉浸深色系、清新明亮系、极简高级系、品牌强化系
  • 每个灵感附带设计拆解笔记——不只是"好看"，要说清楚"为什么好看"
  • 建立配色方案库、排版节奏库、图标风格库、插画风格库
如何落地 How
  • 每周从 Dribbble / Behance / Mobbin / 实际 App 中采集 10+ 案例
  • 每个案例提取：配色值、字号体系、间距规律、动效节奏
  • 用 Figma 建立灵感看板，按标签检索，支持快速调用
  • 结合项目需求，主动输出"风格方向建议"而非等搭档来选
🎯

预期成果 · 当前 0 → 目标 500+ 案例

拿到任何新需求时，30 秒内给出 2-3 个风格方向参考，并说清楚每个方向的优劣和适用场景。设计决策从"我觉得"变成"基于 XX 案例的 XX 模式，因为 XX 原因"。

System 02
✨ 动画体验库 · Motion Library
让动效有章法，不是凭感觉调参数。每个动画都有理由、有标准、有代码。
怎么做 What
  • 整理常用动画模式：转场、微交互、手势反馈、加载态、滚动视差、卡片切换
  • 每个动画定义：参数配置（duration / easing / spring tension & friction）
  • 标注适用场景、性能影响、平台差异（iOS 弹簧 vs Android 贝塞尔）
  • 从已交付项目提炼：极速骑行的卡片扇形展开、蓝莓骑行的 Banner 轮播
如何落地 How
  • 建立动画 Playground——可交互预览每种动效的实际效果
  • 每种动画提供 React Native Animated API 代码片段，可直接复用
  • 参考 Apple HIG Motion 和 Material Motion 建立动效规范
  • 记录踩过的坑：useNativeDriver 限制、Snack 兼容性、性能瓶颈
🎯

预期成果 · 当前 3 种 → 目标 20+ 种动效模式

任何交互需求，直接从库里选动画模式 + 参数，不再每次从零调试。动画丝滑度达到原生 App 水准，60fps 无卡顿。搭档说"动画要丝滑"时，小野能直接给出 3 种方案并说明各自的体感差异。

System 03
🧩 UIKit 组件库 · Design System
统一设计语言，跨项目复用，让公司的每个产品都有一致的品质感。
怎么做 What
  • 从 5 个已交付产品中提炼通用组件：导航栏、卡片、按钮、筛选器、TabBar、渐变、搜索框
  • 定义 Design Token：颜色、字号、间距、圆角、阴影、透明度，统一变量命名
  • 支持品牌主题切换：蓝莓骑行深色系 / 极速骑行运动系 / 通用明亮系
  • 每个组件附带：Props 定义、使用示例、设计规范说明、无障碍要求
如何落地 How
  • 先从高频组件开始：Button → Card → NavBar → TabBar → FilterChip
  • 每个组件提供 React Native / SwiftUI / Flutter 三套实现
  • Figma 端同步维护组件库，设计稿和代码一一对应
  • 建立版本管理，组件更新时自动检测影响范围，防止回归
🎯

预期成果 · 当前零散代码 → 目标 30+ 标准组件

新页面开发效率提升 50%+，不再重复造轮子。任何新项目直接基于 UIKit 搭建，设计一致性从"靠人盯"变成"靠系统保证"。公司内部形成统一的设计语言，品牌辨识度显著提升。

System 04
⚡ 前端技术还原 · Code Fidelity
设计再好，还原不了等于零。这是 UED Leader 和普通设计师的根本区别。
怎么做 What
  • 建立 Figma → Code 的标准化流程：解析设计稿 → 提取 Token → 生成骨架 → 精修细节
  • 掌握多平台技术栈：React Native / SwiftUI / Kotlin Compose / Flutter / Web
  • 深入理解各平台渲染差异：RN 的 Yoga 布局、iOS 的 Auto Layout、Web 的 CSS Grid
  • 建立"无依赖实现"能力：不依赖第三方库实现渐变、动画、手势、复杂布局
如何落地 How
  • 每个 Figma 组件建立"还原检查清单"：尺寸、颜色、字重、间距、圆角、阴影、透明度
  • 交付前自动对比：截图叠加 Figma 设计稿，标注偏差区域
  • 积累平台踩坑手册：Snack Expo 限制、RN 无 LinearGradient 的替代方案、各平台字体渲染差异
  • 持续学习新技术：React Native New Architecture、SwiftUI 新 API、CSS Container Queries
🎯

预期成果 · 当前 ~85% → 目标 95%+ 还原度

Figma 设计稿到代码的还原度稳定在 95%+，不再需要逐像素纠偏。任何平台的技术限制都能找到替代方案，不会因为"技术做不到"而降低设计品质。从"会写代码的设计师"进化为"懂设计的全栈工程师"，真正做到设计即交付。
